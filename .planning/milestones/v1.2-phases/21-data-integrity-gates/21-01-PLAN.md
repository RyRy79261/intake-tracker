<objective>
Create a static schema parser for db.ts and two integrity test suites: one that verifies every Dexie version block includes all tables from prior versions (DATA-04), and one that enforces three-way sync between db.ts tables, BackupData interface keys, and fixture maker functions (DATA-07).

Output: Three files in src/__tests__/integrity/ -- a shared parser utility and two test files that run via `pnpm exec vitest run src/__tests__/integrity/`.

Create `src/__tests__/integrity/parse-schema.ts` that exports:

1. `interface VersionSchema { version: number; tables: string[]; }`
2. `function parseDbSchema(): VersionSchema[]` -- reads `src/lib/db.ts` from disk using `fs.readFileSync`, extracts all `db.version(N).stores({...})` blocks via regex, returns array of `{ version, tables }` sorted by version number ascending.

Implementation details per D-01, D-02, D-03:
- Use `readFileSync` from `"fs"` and `resolve` from `"path"` (Node builtins, available since vitest runs in `environment: "node"`)
- Resolve db.ts path relative to the parser file: `resolve(__dirname, "../../lib/db.ts")`
- Regex pattern: `/db\.version\((\d+)\)\.stores\(\{([^}]+)\}\)/gs` -- the `s` flag for dotAll, `g` for global. The `[^}]+` is safe because Dexie index definition strings never contain `}` (verified against all 6 version blocks).
- Extract table names from each stores block with: `/^\s*(\w+)\s*:/gm` matching the word before the colon on each line.
- Do NOT import db.ts -- that would trigger Dexie initialization side effects. Read the raw file text only.

3. `function getLatestTables(): string[]` -- convenience function that calls `parseDbSchema()`, takes the last entry (highest version), returns its `tables` array. Used by table-sync.test.ts for DATA-07.

Self-validation: after parsing, if `results.length === 0`, throw an error: `"parse-schema: Failed to parse any version blocks from db.ts. Check that db.version(N).stores({...}) pattern is intact."` This catches parser regressions per research open question #1.
