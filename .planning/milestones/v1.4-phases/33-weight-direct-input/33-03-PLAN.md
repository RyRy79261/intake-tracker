Three changes to src/components/ui/inline-edit.tsx:

**Fix 1 — Editing indicator (Gap 1):**
When `isEditing` is true, add a visual underline to the display span to indicate the field is active. Apply a conditional class to the display `<span>` element (line ~115):
- When `isEditing`: add `border-b-2 border-current` (a subtle underline using the current text color)
- This provides a clear but minimal typing indicator without disrupting the layout

Specifically, change the `<span>` from:
```tsx
<span className={displayClassName}>
```
to:
```tsx
<span className={cn(displayClassName, isEditing && "border-b-2 border-current")}>
```
This requires importing/using `cn` which is already imported at line 4.

**Fix 2 — Preserve decimal formatting on focus (Gap 2):**
Change `handleFocus` (line ~69-72) to seed `editValue` using `formatDisplay(value)` instead of `String(value)`:
```tsx
const handleFocus = React.useCallback(() => {
  setIsEditing(true);
  setEditValue(value != null ? formatDisplay(value) : "");
}, [value, formatDisplay]);
```
This ensures 69.00 stays as "69.00" when entering edit mode, because formatDisplay applies `.toFixed(2)`.

**Fix 3 — Guard empty-string display during editing (Gap 3, part 1):**
Change the display ternary (line ~116) from using `||` (logical OR) to a nullish-aware check. The problem: `editValue` can be `""` (falsy) during active typing of intermediate states like "69.", but `||` treats that as falsy and falls through to `formatDisplay(null)` which returns "--".

Change from:
```tsx
{isEditing ? editValue || formatDisplay(null) : formatDisplay(value)}
```
to:
```tsx
{isEditing ? editValue : formatDisplay(value)}
```

When `isEditing` is true and `editValue` is `""`, the display will simply be empty (the user just cleared the field), which is correct UX. When they blur with empty input, `handleBlur` already reverts silently. No need for the `formatDisplay(null)` fallback during active editing.

**Fix Gap 3, part 2 — Input type change:**

In `src/components/weight-card.tsx`, change the InlineEdit props (around line 212) from:
```tsx
type="number"
inputMode="decimal"
step="any"
```
to:
```tsx
type="text"
inputMode="decimal"
pattern="[0-9]*[.]?[0-9]*"
```

Remove the `step="any"` prop entirely (it's only meaningful for `type="number"`).

**Why:** `type="number"` causes browsers to return `e.target.value = ""` for intermediate decimal states like "69." (because "69." is not a valid number). With `type="text"` + `inputMode="decimal"`, mobile still shows the numeric keyboard, but `e.target.value` faithfully reports the string "69." during typing. The `pattern` attribute provides basic client-side validation hint.

The `handleBlur` in inline-edit.tsx already uses `parseFloat()` which correctly handles the final string-to-number conversion, so no other changes are needed.

**E2E test check:** The existing E2E test in `e2e/dashboard.spec.ts` uses `weightInput.fill('71.35')` and `weightInput.focus()` — these work identically with `type="text"` inputs. No E2E changes needed.
