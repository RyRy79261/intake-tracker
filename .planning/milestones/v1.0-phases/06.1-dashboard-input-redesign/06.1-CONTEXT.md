# Phase 06.1: Dashboard Input Redesign - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace popup modals with inline input flows for all dashboard intake metrics (weight, blood pressure, eating, urination, defecation). The water/salt intake card already has a good inline pattern — other metrics should follow its lead. Edit dialogs are kept only for editing previously entered records. No new data types or schema changes.

</domain>

<decisions>
## Implementation Decisions

### Input pattern per metric
- **Water/salt**: Keep as-is — already has inline increment/decrement with coffee/juice/food sub-types. This is the reference pattern.
- **Weight**: Replace text input with centered value display flanked by minus (left) and plus (right) buttons. Pre-fill with most recent recorded weight. Save button below. Increment step configurable (default 0.1kg).
- **Blood pressure**: Inline systolic/diastolic/HR fields directly in the card — no popup dialog for new entries. Position/arm/irregular toggles available via expandable detail section.
- **Eating**: Quick "I ate" one-tap log button on the card. Optional expandable section for note and grams — not a modal.
- **Urination**: Inline amount selector (the existing amount options) directly in the card. Quick log on selection. Optional note via expansion.
- **Defecation**: Inline Bristol scale selector directly in the card. Quick log on selection. Optional note via expansion.

### Weight-specific UX
- Left-hand minus button, right-hand plus button (user specified)
- Large centered weight display showing the value
- Pre-filled with the most recent recorded weight value
- Save/confirm button below the +/- row
- Step size from settings store (currently uses 0.1kg increments)

### Quick-log pattern
- Primary action for each metric is ONE tap (or one tap + confirm for weight)
- Detail fields (note, time override, advanced options) available via expandable section — not a mandatory step
- Cards should feel as fast as the water card to use

### Edit vs new entry
- New entry: always inline within the card — no modal/dialog
- Edit existing record: keep the existing edit dialog pattern (EditWeightDialog, EditBloodPressureDialog, etc.)
- This is a hard rule — modals only for editing, never for initial data entry

### Claude's Discretion
- Exact layout/spacing of the inline inputs
- Animation for expandable detail sections (can follow existing framer-motion patterns)
- Whether to show a success toast or inline confirmation after logging
- How to handle the transition from old dialog-based to new inline pattern (can be direct replacement)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `IntakeCard` (intake-card.tsx): The gold-standard inline pattern — increment/decrement, liquid type tabs, coffee presets, manual input. Other cards should feel equally fast.
- `RecentEntriesList` component: Already used by all cards for showing recent entries with edit/delete actions.
- `CollapsibleTimeInputControlled`: Existing expandable time override pattern, reusable for detail sections.
- `useEditRecord` hook: Shared hook for edit dialog state management — stays as-is.
- `useDeleteWithToast` hook: Shared hook for delete with undo toast — stays as-is.
- `CARD_THEMES`: Theming system for card colors/icons — stays as-is.

### Established Patterns
- All cards follow: themed header → input area → submit → recent entries list → edit dialog
- Zustand settings store for user preferences (increments, defaults)
- React Query mutations for optimistic updates
- Zod validation for form schemas

### Integration Points
- Cards are rendered in `src/app/page.tsx` main dashboard
- Each card has its own query hooks in `src/hooks/use-*-queries.ts`
- Card themes in `src/lib/card-themes.ts`
- Constants (urination amounts, BP categories, etc.) in `src/lib/constants.ts`

</code_context>
