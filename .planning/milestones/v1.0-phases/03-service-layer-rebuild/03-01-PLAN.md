  <name>Task 1: Timezone utility module + syncFields update</name>
  <files>src/lib/timezone.ts, src/lib/utils.ts</files>
  <action>
Create `src/lib/timezone.ts` with these exports:

1. `getDeviceTimezone(): string` — returns `Intl.DateTimeFormat().resolvedOptions().timeZone`. Cache the result for SSR safety (return "UTC" if `typeof window === "undefined"`).

2. `localTimeToUTCMinutes(hours: number, minutes: number, timezone: string): number` — convert a local HH:MM in a given IANA timezone to minutes-from-midnight-UTC. Use `Intl.DateTimeFormat` to determine UTC offset for the timezone. Wrap result with `((result % 1440) + 1440) % 1440` to handle negative values.

3. `utcMinutesToLocalTime(utcMinutes: number, timezone: string): { hours: number; minutes: number }` — reverse conversion. Also provide a `formatLocalTime(utcMinutes: number, timezone: string): string` that returns "HH:MM" formatted string.

4. `localHHMMStringToUTCMinutes(timeStr: string, timezone: string): number` — convenience wrapper that parses "HH:MM" string and calls localTimeToUTCMinutes.

5. `MIGRATION_TIMEZONE_CUTOFF = new Date("2026-02-12T00:00:00Z").getTime()` — exported constant for migration use.

6. `getTimezoneForTimestamp(timestamp: number): string` — returns "Africa/Johannesburg" if timestamp < MIGRATION_TIMEZONE_CUTOFF, "Europe/Berlin" otherwise. Used during v11 migration backfill.

Update `src/lib/utils.ts` `syncFields()` to include `timezone: getDeviceTimezone()` in the returned object. Import getDeviceTimezone from timezone.ts.

**db.ts changes:**

1. Add `timezone: string` field to ALL record interfaces: IntakeRecord, WeightRecord, BloodPressureRecord, EatingRecord, UrinationRecord, DefecationRecord, DoseLog, DailyNote, AuditLog. Make it required (not optional) — migration backfills it.

2. Update PhaseSchedule interface:
   - Add `scheduleTimeUTC: number` (minutes from midnight UTC, integer)
   - Add `anchorTimezone: string` (IANA timezone when schedule was created)
   - Keep `time: string` as `@deprecated` (will exist in DB records from v10, migration copies it)

3. Add `timezone` to InventoryItem and InventoryTransaction interfaces.

4. Add new AuditAction values if missing: `"dose_untaken"`, `"prescription_deleted"`, `"phase_completed"`, `"phase_started"`, `"stock_recalculated"`, `"inventory_added"`, `"inventory_deleted"`.

5. Add `db.version(11)` with ALL v10 store definitions repeated (Dexie requirement). No index changes needed — timezone is not indexed. The store definitions are identical to v10.

6. Add `.upgrade()` to v11 that:
   - Backfills `timezone` on all record tables using `getTimezoneForTimestamp(record.timestamp ?? record.createdAt)` from timezone.ts
   - Converts PhaseSchedule `time` (HH:MM string) to `scheduleTimeUTC` integer using `localHHMMStringToUTCMinutes(record.time, getTimezoneForTimestamp(record.createdAt))`. Set `anchorTimezone` to the same timezone used for conversion
   - Sets `updatedAt = Date.now()` on modified records

**audit-service.ts:**

Create `src/lib/audit-service.ts` with:

1. `buildAuditEntry(action: AuditAction, details: Record<string, unknown>): AuditLog` — builds a complete audit log record using syncFields() + `timestamp: Date.now()`, details serialized as JSON string.

2. `writeAuditLog(action: AuditAction, details: Record<string, unknown>): Promise<void>` — calls buildAuditEntry and adds to db.auditLogs. This is the standalone version for non-transactional use.

3. Re-export AuditAction type from db.ts for convenience.

Note: Inside transactions, callers will use `buildAuditEntry` to create the record and `db.auditLogs.add()` directly (to stay within the transaction scope). `writeAuditLog` is for standalone audit entries outside transactions.
