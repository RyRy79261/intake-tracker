<decisions>
## Implementation Decisions

### Design Token Strategy
- Promote ALL domain colors to CSS custom properties in globals.css (currently only water and salt have them)
- Domains to tokenize: water, salt, weight, BP, eating, urination, defecation, caffeine, alcohol, medication
- Keep current Tailwind hues for parity: water=sky, salt=amber, weight=emerald, BP=rose, eating=orange, caffeine=yellow, alcohol=fuchsia, medication=teal
- Same hue in light/dark mode with lightness/saturation adjustments for readability
- Pencil variables must mirror these CSS custom properties exactly
- Light and dark theme variants in Pencil matching globals.css `:root` and `.dark` definitions

### Component Library — Full Scope
- **All 17 shadcn primitives** mirrored in Pencil: button, card, collapsible, dialog, drawer, input, label, numeric-input, progress, scroll-area, select, sheet, switch, tabs, textarea, toast, toaster
- **All domain-specific components**: intake card (water/salt), health metric card (BP/weight), tracking card (eating/urination/defecation), caffeine card, alcohol card, substance type picker, prescription card, dose logger, inventory widget
- **App chrome**: app header, quick-nav footer
- All components match current code implementation exactly — 1:1 parity, no redesign

### Screen Designs — Comprehensive Coverage
- **4 main screens**: Dashboard (`/`), Medications (`/medications`), Analytics (`/analytics`), Settings (`/settings`)
- **All states per screen**: populated, empty, loading skeleton, error
- **Analytics tabs**: All 4 as separate screen variants — Records, Insights, Correlations, Titration (populated state)
- **Full medication flows**: list view, detail view, add-prescription wizard steps, dose logging interaction, inventory management
- **All overlay dialogs/drawers**: edit intake, add/edit BP, add/edit weight, edit eating, edit urination, edit defecation, food calculator, manual input, history drawer, settings drawer, about dialog, pin dialog, substance type picker, voice input
- All screens match current deployed app — parity capture, not redesign

### Pencil Workflow (Design-First)
- **Design-first before coding** — each future phase starts with Pencil designs reviewed/approved, then code implements
- **Pencil always leads** — when a phase changes existing screens, update the Pencil design first, then implement in code
- **Single file**: everything in `design/app.pen` — tokens, components, and all screen designs
- **Document in CLAUDE.md** — add a Pencil workflow section covering design-first convention, how to use Pencil for a phase, naming conventions, update process

### Claude's Discretion
- Internal organization within app.pen (page/frame structure, naming conventions)
- Component variant naming in Pencil
- How to represent loading skeleton and error states visually
- Chart/graph representation in Analytics screen designs
- Level of detail in wizard step designs

</decisions>
