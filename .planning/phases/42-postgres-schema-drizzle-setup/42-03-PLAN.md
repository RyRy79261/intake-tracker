```typescript
export interface DexieTableSchema {
  /** Dexie table name, camelCase (e.g., "intakeRecords") — matches the key in db.version(15).stores({...}) */
  tableName: string;
  /** TypeScript interface name (e.g., "IntakeRecord") — matches the exported `interface` in db.ts */
  interfaceName: string;
  /** Property names declared on the interface, including optional ones (?). camelCase. */
  fields: string[];
}

/**
 * Walk src/lib/db.ts using the TypeScript compiler API and return one
 * DexieTableSchema per table declared in db.version(15).stores({...}).
 *
 * Order of operations:
 *   1. Parse db.ts into a TypeScript SourceFile
 *   2. Find the call expression `db.version(<highest>).stores({...})` — the object literal
 *      keys ARE the canonical Dexie table list
 *   3. For each key, locate the matching EntityTable<X, "id"> type in the Dexie declaration
 *      (or the interface X by convention: intakeRecords → IntakeRecord, etc.)
 *   4. Walk the interface's property signatures and extract the property names
 *
 * The mapping from table name to interface name follows one rule:
 *   tableName = camelCaseWithPluralSuffix → interfaceName = PascalCaseSingular
 *   Examples:
 *     intakeRecords → IntakeRecord
 *     weightRecords → WeightRecord
 *     bloodPressureRecords → BloodPressureRecord
 *     prescriptions → Prescription
 *     medicationPhases → MedicationPhase
 *     phaseSchedules → PhaseSchedule
 *     inventoryItems → InventoryItem
 *     inventoryTransactions → InventoryTransaction
 *     dailyNotes → DailyNote
 *     doseLogs → DoseLog
 *     auditLogs → AuditLog
 *     substanceRecords → SubstanceRecord
 *     titrationPlans → TitrationPlan
 *     urinationRecords → UrinationRecord
 *     defecationRecords → DefecationRecord
 *     eatingRecords → EatingRecord
 *
 * The walker derives this mapping via a small helper (drop trailing 's' or 'es',
 * capitalize first letter). It validates that the resolved interface name exists
 * as an exported type in the source file; if not, it throws a clear error.
 */
export function extractDexieSchema(dbTsPath?: string): DexieTableSchema[];
```

    **Implementation outline:**

    ```typescript
    import * as fs from "node:fs";
    import * as path from "node:path";
    import ts from "typescript";

    export interface DexieTableSchema {
      tableName: string;
      interfaceName: string;
      fields: string[];
    }

    /**
     * Static mapping from Dexie table name (camelCase) to TypeScript interface name
     * (PascalCase singular). Explicit to avoid fragile pluralization heuristics.
     * Must be kept in sync with src/lib/db.ts — if the parity test fails loudly
     * when a new table is added without updating this map, that's a feature.
     */
    const TABLE_TO_INTERFACE: Record<string, string> = {
      intakeRecords: "IntakeRecord",
      weightRecords: "WeightRecord",
      bloodPressureRecords: "BloodPressureRecord",
      eatingRecords: "EatingRecord",
      urinationRecords: "UrinationRecord",
      defecationRecords: "DefecationRecord",
      substanceRecords: "SubstanceRecord",
      prescriptions: "Prescription",
      medicationPhases: "MedicationPhase",
      phaseSchedules: "PhaseSchedule",
      inventoryItems: "InventoryItem",
      inventoryTransactions: "InventoryTransaction",
      doseLogs: "DoseLog",
      dailyNotes: "DailyNote",
      auditLogs: "AuditLog",
      titrationPlans: "TitrationPlan",
    };

    const DB_TS_DEFAULT_PATH = path.resolve(__dirname, "../../src/lib/db.ts");

    export function extractDexieSchema(dbTsPath: string = DB_TS_DEFAULT_PATH): DexieTableSchema[] {
      const source = fs.readFileSync(dbTsPath, "utf-8");
      const sourceFile = ts.createSourceFile(
        dbTsPath,
        source,
        ts.ScriptTarget.Latest,
        /* setParentNodes */ true,
        ts.ScriptKind.TS
      );

      // Step 1: Collect every exported interface declaration's property names.
      // Map: interfaceName -> string[] of property names (including optional ones,
      //                                                   excluding deprecated-only shapes)
      const interfaceFields = new Map<string, string[]>();
      ts.forEachChild(sourceFile, function walk(node) {
        if (ts.isInterfaceDeclaration(node)) {
          const name = node.name.text;
          const fields: string[] = [];
          for (const member of node.members) {
            if (ts.isPropertySignature(member) && member.name && ts.isIdentifier(member.name)) {
              fields.push(member.name.text);
            }
          }
          interfaceFields.set(name, fields);
        }
        ts.forEachChild(node, walk);
      });

      // Step 2: Validate the static mapping — every expected interface must exist.
      const result: DexieTableSchema[] = [];
      for (const [tableName, interfaceName] of Object.entries(TABLE_TO_INTERFACE)) {
        const fields = interfaceFields.get(interfaceName);
        if (!fields) {
          throw new Error(
            `Cannot resolve Dexie interface for table '${tableName}' — ` +
            `expected interface '${interfaceName}' not found in ${dbTsPath}`
          );
        }
        result.push({ tableName, interfaceName, fields });
      }

      return result;
    }
    ```

    **Key design decisions:**

    1. **Static mapping over pluralization heuristics.** PhaseSchedule, DoseLog, AuditLog, DailyNote, MedicationPhase — the English pluralization rules are inconsistent enough that a regex-based approach would have edge cases. The static mapping makes the contract explicit and fails loudly when someone adds a new table without updating it.

    2. **Use `ts.createSourceFile` not `ts.createProgram`.** `createProgram` requires a full tsconfig and module resolution — slow and has more failure modes. `createSourceFile` parses a single file into an AST, which is all we need. ~5x faster.

    3. **Walk top-level only, then recurse only for interface discovery.** The walker is rooted at `sourceFile` and descends via `ts.forEachChild`. Because all the interfaces in db.ts are top-level exports, the walker finds them on the first level. The recursion is present as a safety net but is not strictly necessary.

    4. **Include optional fields.** `note?: string` is still a field that must exist in the Drizzle table (as nullable). The extractor records optional fields in the returned array without any optionality marker — the parity test only cares about presence.

    5. **Exclude deprecated getter/setter patterns.** If db.ts ever introduces a getter like `get age()` instead of a property, the walker skips it (only `ts.PropertySignature` is collected). This is fine — deprecated `currentStock` and deprecated `time` are declared as normal `?` properties, so they are collected.

    6. **Import `typescript` as a runtime dependency in tests.** TypeScript is already in `devDependencies` (via Next.js transitive dep, or explicitly). Confirm by grepping package.json: `grep -q '"typescript"' package.json`. If it's in devDependencies, tests can import it because vitest runs in the Node toolchain that has devDeps available.

    7. **No ts-morph, no expect-type.** Both add new deps for zero benefit over the built-in compiler API.

    ```typescript
    /**
     * Schema parity gate — Dexie ↔ Drizzle drift detector.
     *
     * This test runs as part of `pnpm test` and therefore in every CI run. It fails
     * when Dexie (src/lib/db.ts v15 interfaces) and Drizzle (src/db/schema.ts) fall
     * out of sync — which is the moment a future phase adds a field to one side
     * without the other.
     *
     * The comparator is STRUCTURAL (field name presence) — not type-based. Union
     * types in Dexie and text+CHECK in Drizzle are considered equivalent because
     * both encode the same set of values. If this decision ever reverses, switch
     * to drizzle-zod + expect-type for compile-time type comparison.
     *
     * The only permitted Drizzle-only column is `userId` (single-user Dexie
     * has no user ownership field — see 42-CONTEXT.md D-12).
     */
    import { describe, it, expect } from "vitest";
    import { getTableColumns, Table } from "drizzle-orm";
    import * as drizzleSchema from "@/db/schema";
    import { extractDexieSchema, type DexieTableSchema } from "./dexie-schema-extractor";

    const DEXIE_TABLES: DexieTableSchema[] = extractDexieSchema();

    // Fields that exist on the Drizzle side but must NEVER match a Dexie field —
    // the exemption list for the "no extra Drizzle columns" check.
    const DRIZZLE_ONLY_EXEMPTIONS = new Set<string>(["userId"]);

    function getDrizzleTable(tableName: string): Table | undefined {
      // drizzleSchema is a Record<string, Table> at runtime; TS sees it as the
      // module shape. Cast through unknown to bypass the keyof constraint.
      return (drizzleSchema as unknown as Record<string, Table>)[tableName];
    }

    function getDrizzleColumnNames(table: Table): string[] {
      return Object.keys(getTableColumns(table));
    }

    describe("Dexie ↔ Drizzle schema parity", () => {
      it("extracts all 16 Dexie tables from src/lib/db.ts", () => {
        expect(DEXIE_TABLES).toHaveLength(16);
        const names = DEXIE_TABLES.map((t) => t.tableName);
        expect(names).toContain("intakeRecords");
        expect(names).toContain("substanceRecords");
        expect(names).toContain("titrationPlans");
      });

      it.each(DEXIE_TABLES)(
        "Dexie table '$tableName' has a matching Drizzle export",
        ({ tableName }) => {
          const drizzleTable = getDrizzleTable(tableName);
          expect(drizzleTable, `Missing Drizzle export: drizzleSchema.${tableName}`).toBeDefined();
        }
      );

      it.each(DEXIE_TABLES)(
        "Dexie table '$tableName': every field is present in the Drizzle table",
        ({ tableName, fields }) => {
          const drizzleTable = getDrizzleTable(tableName);
          if (!drizzleTable) {
            throw new Error(`Drizzle table '${tableName}' not found — see previous test`);
          }
          const drizzleCols = new Set(getDrizzleColumnNames(drizzleTable));
          const missing = fields.filter((f) => !drizzleCols.has(f));
          expect(
            missing,
            `${tableName}: Dexie fields missing from Drizzle: [${missing.join(", ")}]. ` +
              `Drizzle has [${Array.from(drizzleCols).join(", ")}]`
          ).toEqual([]);
        }
      );

      it.each(DEXIE_TABLES)(
        "Dexie table '$tableName': Drizzle has no extra columns beyond userId",
        ({ tableName, fields }) => {
          const drizzleTable = getDrizzleTable(tableName);
          if (!drizzleTable) {
            throw new Error(`Drizzle table '${tableName}' not found`);
          }
          const dexieFieldSet = new Set(fields);
          const extras = getDrizzleColumnNames(drizzleTable).filter(
            (col) => !dexieFieldSet.has(col) && !DRIZZLE_ONLY_EXEMPTIONS.has(col)
          );
          expect(
            extras,
            `${tableName}: Drizzle has columns not in Dexie (only 'userId' is exempt): [${extras.join(", ")}]`
          ).toEqual([]);
        }
      );

      it.each(DEXIE_TABLES)(
        "Dexie table '$tableName': Drizzle has a userId column",
        ({ tableName }) => {
          const drizzleTable = getDrizzleTable(tableName);
          if (!drizzleTable) {
            throw new Error(`Drizzle table '${tableName}' not found`);
          }
          const cols = getDrizzleColumnNames(drizzleTable);
          expect(cols, `${tableName}: missing required 'userId' column`).toContain("userId");
        }
      );
    });
    ```

    **Test design notes:**

    1. **`it.each`** — parameterized tests give a per-table failure report, which makes drift root-cause obvious. A single aggregate `describe.each` over all tables is harder to read in CI output.

    2. **`DEXIE_TABLES` computed at module load** — the extractor runs once at test-file evaluation, not once per test. Zero IO redundancy.

    3. **`getTableColumns(table)` returns TS property names, not SQL column names.** This is the critical alignment with the extractor (which also returns TS property names from the interface). The comparison is apples-to-apples camelCase.

    4. **Cast through `unknown`.** `drizzleSchema` is typed as a module shape with specific exports, so indexing with a dynamic string trips the TS checker. `as unknown as Record<string, Table>` is the cleanest escape hatch. Vitest doesn't care — this is test-only code.

    5. **Fail early if a test can't find its table.** The second test asserts existence; later tests short-circuit via `throw new Error` if a table is missing. This prevents cascading failures that obscure the real issue.

    6. **Do NOT strictly compare column types.** Dexie `note?: string` and Drizzle `note: text("note")` nullable are "equivalent" for this test's purpose. The comparator is field-presence-only, deliberately — see the header comment.

    7. **Do NOT check CHECK constraints or indexes.** The parity test's job is "every field exists on both sides." CHECK constraints and indexes are schema-level invariants verified by the migration review (Plan 42-02) and by runtime integration tests in Phase 43. Out of scope here.
