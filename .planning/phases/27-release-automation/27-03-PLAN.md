# Plan 27-03: Release Please Workflow & Cleanup

<objective>
Create the Release Please GitHub Actions workflow and configuration, delete the old version-bump.yml, so that merging conventional commits to main automatically produces versioned releases with changelogs and GitHub Releases.
</objective>

<threat_model>
## Threat Model

| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-27-03-01 | Release Please workflow has insufficient permissions, fails silently | Medium | Explicitly declare `contents: write` and `pull-requests: write` permissions in workflow |
| T-27-03-02 | Deleting version-bump.yml before Release Please is ready causes gap in version management | Low | Wave 2 dependency ensures version bootstrap (Plan 01) and commit hooks (Plan 02) are in place first |
| T-27-03-03 | CHANGELOG.md includes noise commits (docs, chore, etc.) | Low | Explicitly configure `changelog-sections` in release-please-config.json to only include feat and fix |
| T-27-03-04 | bootstrap-sha mismatch causes Release Please to include all historical commits | Medium | Belt-and-suspenders: both v1.2.0 tag (Plan 01) AND bootstrap-sha in config anchor the starting point |

No high-severity threats. All mitigations are implemented in task actions.
</threat_model>

<action>
Create `release-please-config.json` at the repo root with this exact content:

```json
{
  "release-type": "node",
  "changelog-sections": [
    { "type": "feat", "section": "Features" },
    { "type": "fix", "section": "Bug Fixes" }
  ],
  "bootstrap-sha": "a3a0b2d0db39613fe964d7f58137f8063768ef77",
  "packages": {
    ".": {}
  }
}
```

This configuration:
- `release-type: node` — tells Release Please to update `package.json` version
- `changelog-sections` — restricts CHANGELOG.md to only `feat` and `fix` entries (D-06), keeping the changelog clean and user-focused
- `bootstrap-sha` — tells Release Please to ignore all commits before this SHA (belt-and-suspenders with the v1.2.0 tag from Plan 01)
- `packages: { ".": {} }` — single-package mode pointing to repo root
</action>

<task id="27-03-02">
<title>Create Release Please GitHub Actions workflow</title>

<read_first>
- `.github/workflows/ci.yml` (existing CI workflow — understand patterns: runner, node version, pnpm setup)
- `.github/workflows/version-bump.yml` (workflow being replaced — understand trigger pattern)
- `.planning/phases/27-release-automation/27-RESEARCH.md` (section 2.2 for workflow format)
</read_first>

<action>
Create `.github/workflows/release-please.yml` with this exact content:

```yaml
name: Release Please

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

jobs:
  release-please:
    runs-on: ubuntu-latest
    steps:
      - uses: googleapis/release-please-action@v4
        with:
          release-type: node
```

This workflow:
- Triggers on every push to `main` (same trigger as the old version-bump.yml)
- Has `contents: write` permission for creating tags and GitHub Releases
- Has `pull-requests: write` permission for creating/updating the release PR
- Uses `release-type: node` which reads `release-please-config.json` and `.release-please-manifest.json` automatically
- Uses the default `GITHUB_TOKEN` (no custom token needed)
- Is intentionally minimal — Release Please reads its config from the repo files, not workflow inputs
</action>

<acceptance_criteria>
- File `.github/workflows/release-please.yml` exists
- File contains `googleapis/release-please-action@v4`
- File contains `permissions:` block with `contents: write` and `pull-requests: write`
- File triggers on `push` to `main` branch
- YAML is valid (no syntax errors)
</acceptance_criteria>
</task>
