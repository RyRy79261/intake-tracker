# Plan 27-01: Version Bootstrap & Historical Anchor

<objective>
Reconcile package.json version from `0.1.0` to `1.2.0` and create the historical git tag and Release Please manifest so Release Please has a clean starting point.
</objective>

<threat_model>
## Threat Model

| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-27-01-01 | Incorrect version anchor causes Release Please to miscompute next version | Medium | Belt-and-suspenders: both v1.2.0 tag AND manifest file pin the version |
| T-27-01-02 | Tag at wrong commit includes unrelated changes in first release | Low | Verified commit a3a0b2d is exactly "chore: complete v1.2 CI & Data Integrity milestone" |

No high-severity threats. All mitigations are implemented in task actions.
</threat_model>

## Tasks

<task id="27-01-01">
<title>Update package.json version to 1.2.0</title>

<read_first>
- `package.json` (current version field is `0.1.0`)
</read_first>

<action>
Edit `package.json` to change the `"version"` field from `"0.1.0"` to `"1.2.0"`.

This is a single field change — do not modify any other fields. The version `1.2.0` aligns with the last shipped milestone (v1.2 CI & Data Integrity).
</action>

<acceptance_criteria>
- `node -p "require('./package.json').version"` outputs exactly `1.2.0`
- No other fields in `package.json` are modified (diff shows only the version line)
</acceptance_criteria>
</task>

<task id="27-01-02">
<title>Create Release Please manifest file</title>

<read_first>
- `.release-please-manifest.json` (should not exist yet)
- `.planning/phases/27-release-automation/27-RESEARCH.md` (section 2.3 for manifest format)
</read_first>

<action>
Create `.release-please-manifest.json` at the repo root with this exact content:

```json
{
  ".": "1.2.0"
}
```

This tells Release Please the current version is `1.2.0`. The first release will compute the next version from conventional commits after the `v1.2.0` tag.
</action>

<acceptance_criteria>
- File `.release-please-manifest.json` exists at repo root
- `node -e "const m=JSON.parse(require('fs').readFileSync('.release-please-manifest.json','utf8')); console.log(m['.'])"` outputs `1.2.0`
- File contains valid JSON (no trailing commas, correct structure)
</acceptance_criteria>
</task>

<task id="27-01-03">
<title>Create v1.2.0 git tag at milestone completion commit</title>

<read_first>
- Output of `git log --oneline a3a0b2d -1` to verify the commit message is "chore: complete v1.2 CI & Data Integrity milestone"
- Output of `git tag -l` to confirm v1.2.0 does not already exist
</read_first>

<action>
Create an annotated git tag `v1.2.0` pointing at commit `a3a0b2d`:

```bash
git tag -a v1.2.0 a3a0b2d -m "v1.2.0 — CI & Data Integrity milestone"
```

This provides the historical anchor that Release Please uses to determine which commits are new since the last release. Do NOT push the tag yet — it will be pushed when the branch is merged or manually pushed later.

**Important:** Verify the commit SHA first. Run `git log --oneline a3a0b2d -1` and confirm it shows "chore: complete v1.2 CI & Data Integrity milestone". If the SHA is wrong, check `.planning/STATE.md` for the correct completion commit.
</action>

<acceptance_criteria>
- `git tag -l v1.2.0` outputs `v1.2.0`
- `git log --oneline v1.2.0 -1` outputs a line containing `a3a0b2d` and "complete v1.2"
- Tag is annotated (not lightweight): `git cat-file -t v1.2.0` outputs `tag`
</acceptance_criteria>
</task>

```bash
# Tag target check

```
</verification>
