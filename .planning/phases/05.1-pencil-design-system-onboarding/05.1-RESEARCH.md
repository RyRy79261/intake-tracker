# Phase 5.1: Pencil Design System Onboarding - Research

**Researched:** 2026-03-10
**Domain:** Pencil.dev design system, CSS design tokens, design-code parity
**Confidence:** HIGH

## Summary

This phase onboards the Pencil.dev design system by: (1) promoting all domain colors to CSS custom properties, (2) mirroring those tokens as Pencil variables with light/dark theme support, (3) creating reusable Pencil components matching all 17 shadcn primitives and all domain-specific components, (4) designing all app screens with state variants, and (5) documenting the design-first workflow in CLAUDE.md.

The existing `design/app.pen` file has a single frame (Intake Dashboard), 24 color variables (no theme axis), and zero reusable components. It needs substantial expansion. The `.pen` format is JSON-based with a well-documented node tree structure supporting frames, text, icons (Lucide), layout (flexbox-style), variables with theme-conditional values, and reusable components via the `reusable: true` flag plus `ref` type instances.

**Primary recommendation:** Work through the Pencil MCP tools (`set_variables`, `batch_design`, `get_screenshot`) to build everything in `design/app.pen`. Start with design tokens and themes, then primitives, then domain components, then full screen compositions. All work targets the single `design/app.pen` file.

## Architecture Patterns

### Recommended app.pen Organization

The `.pen` file is a flat JSON tree. Use top-level frames as "pages" to organize content logically.

```
design/app.pen (single file)
  variables: { all design tokens with theme-conditional values }
  themes: { mode: ["light", "dark"] }
  children:
    Frame: "Tokens Reference"          # Visual reference of all tokens
    Frame: "Components / Primitives"   # shadcn primitives (reusable: true)
    Frame: "Components / Domain"       # Domain-specific components (reusable: true)
    Frame: "Components / Chrome"       # App header, quick-nav footer (reusable: true)
    Frame: "Screen / Dashboard"        # Populated state
    Frame: "Screen / Dashboard Empty"  # Empty state
    Frame: "Screen / Dashboard Loading"# Skeleton state
    Frame: "Screen / Dashboard Error"  # Error state
    Frame: "Screen / Medications"      # List view populated
    Frame: "Screen / Medications Empty"
    ... (repeat for all screens and states)
    Frame: "Overlay / Edit Intake"     # Dialog/drawer designs
    Frame: "Overlay / Edit BP"
    ... (all overlays)
    Frame: "Flow / Add Prescription"   # Wizard steps
```

### Pattern 1: Design Token Architecture

**What:** CSS custom properties in globals.css become the single source of truth. Pencil variables mirror them. Tailwind consumes them.

**Flow:** `globals.css (source) -> tailwind.config.ts (consumption) -> Pencil variables (mirror)`

**CSS Custom Property Pattern (existing for water/salt, extend to all domains):**
```css
:root {
  --water: 200 85% 55%;
  --water-foreground: 0 0% 100%;
  --salt: 30 80% 55%;
  --salt-foreground: 0 0% 100%;
  /* NEW: all domain colors */
  --weight: 160 84% 39%;           /* emerald-500 */
  --weight-foreground: 0 0% 100%;
  --bp: 350 89% 60%;               /* rose-500 */
  --bp-foreground: 0 0% 100%;
  --eating: 25 95% 53%;             /* orange-500 */
  --eating-foreground: 0 0% 100%;
  --urination: 280 68% 60%;         /* violet-500 approx */
  --urination-foreground: 0 0% 100%;
  --defecation: 25 25% 45%;         /* stone-600 approx */
  --defecation-foreground: 0 0% 100%;
  --caffeine: 48 96% 53%;           /* yellow-400 */
  --caffeine-foreground: 0 0% 0%;
  --alcohol: 292 84% 61%;           /* fuchsia-500 */
  --alcohol-foreground: 0 0% 100%;
  --medication: 168 76% 36%;        /* teal-600 */
  --medication-foreground: 0 0% 100%;
}
```

**Tailwind config extension:**
```typescript
// Add to tailwind.config.ts colors:
weight: {
  DEFAULT: "hsl(var(--weight))",
  foreground: "hsl(var(--weight-foreground))",
},
bp: {
  DEFAULT: "hsl(var(--bp))",
  foreground: "hsl(var(--bp-foreground))",
},
// ... same pattern for all domains
```

### Pattern 2: Pencil Theme-Conditional Variables

**What:** Pencil supports theme axes with conditional variable values, matching CSS light/dark modes.

**Pencil format:**
```json
{
  "themes": {
    "mode": ["light", "dark"]
  },
  "variables": {
    "water": {
      "type": "color",
      "value": [
        { "theme": { "mode": "light" }, "value": "#38BDF8" },
        { "theme": { "mode": "dark" }, "value": "#0EA5E9" }
      ]
    },
    "water-bg": {
      "type": "color",
      "value": [
        { "theme": { "mode": "light" }, "value": "#F0F9FF" },
        { "theme": { "mode": "dark" }, "value": "#0C1929" }
      ]
    }
  }
}
```

### Pattern 3: Reusable Components

**What:** Mark frames as `reusable: true`, then instantiate with `ref` type nodes.

**Creating a reusable component:**
```json
{
  "type": "frame",
  "id": "btn-primary",
  "name": "Button / Primary",
  "reusable": true,
  "width": "fit_content",
  "height": 40,
  "fill": "$primary",
  "cornerRadius": "$radius",
  "padding": [0, 16],
  "justifyContent": "center",
  "alignItems": "center",
  "children": [
    {
      "type": "text",
      "id": "btn-primary-label",
      "content": "Button",
      "fill": "$primary-foreground",
      "fontSize": 14,
      "fontWeight": "500"
    }
  ]
}
```

**Using a component instance:**
```json
{
  "type": "ref",
  "id": "instance-1",
  "ref": "btn-primary",
  "x": 100,
  "y": 200,
  "descendants": {
    "btn-primary-label": {
      "content": "Save Changes"
    }
  }
}
```

### Pattern 4: Screen Frame Convention

**What:** Each screen design uses a 393px width frame (iPhone 14 Pro logical width) matching the mobile-first max-w-lg constraint.

```json
{
  "type": "frame",
  "name": "Screen / Dashboard",
  "width": 393,
  "height": 852,
  "fill": "$bg",
  "layout": "vertical",
  "children": [
    { "type": "ref", "ref": "app-header-id" },
    { "comment": "... screen content using component refs ..." },
    { "type": "ref", "ref": "quick-nav-footer-id" }
  ]
}
```

### Anti-Patterns to Avoid
- **Hardcoded colors in designs:** Always use `$variable-name` references, never hex values directly in component/screen frames. The existing app.pen has some hardcoded hex values (e.g., `#EFF6FF`) that should be replaced with variable references.
- **Non-reusable components:** Every component that appears on multiple screens MUST be marked `reusable: true` and instantiated via `ref`. Copy-pasting creates maintenance nightmares.
- **Missing states:** Every screen needs populated, empty, loading, and error variants. Skipping states means future phases lack design reference.
- **Wrong font family:** The app uses Outfit font (via next/font), not Inter. The existing app.pen incorrectly uses Inter -- must be corrected to Outfit.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Design token sync | Manual hex value copying between CSS and Pencil | Pencil MCP `set_variables` tool | Ensures exact parity, catches drift |
| Component creation | Direct JSON editing with manual ID generation | Pencil MCP `batch_design` tool | Handles unique IDs, validates structure |
| Visual verification | Relying on JSON inspection alone | Pencil MCP `get_screenshot` tool | Catches layout/spacing issues JSON can't reveal |
| Icon rendering | Custom SVG paths | Pencil `icon_font` type with Lucide family | App uses Lucide icons; Pencil has native Lucide support |

**Key insight:** The Pencil MCP tools abstract away the `.pen` JSON complexity. Use them rather than editing JSON directly, even though the format is readable.

## Common Pitfalls

### Pitfall 1: Font Family Mismatch
**What goes wrong:** Designs use Inter (Pencil default) instead of Outfit (app's actual font).
**Why it happens:** The existing app.pen already has this wrong (uses Inter throughout).
**How to avoid:** Set fontFamily to "Outfit" on all text nodes. Create a text style variable if Pencil supports it, otherwise enforce in every text node.
**Warning signs:** Text in screenshots looks different from the running app.

### Pitfall 2: Hardcoded Colors Instead of Variables
**What goes wrong:** Hex colors used directly instead of `$variable` references. Changes to tokens don't propagate.
**Why it happens:** Quick prototyping uses direct colors; existing app.pen has hardcoded values like `#EFF6FF`.
**How to avoid:** Define ALL colors as variables first, then reference exclusively via `$name` syntax.
**Warning signs:** `get_screenshot` after variable change shows no visual update.

### Pitfall 3: Missing Theme Axis
**What goes wrong:** Variables have single values, no light/dark variants. Dark mode designs can't be previewed.
**Why it happens:** Theme axis (`themes: { mode: ["light", "dark"] }`) must be explicitly defined at document root.
**How to avoid:** Define theme axis before creating any theme-conditional variables.
**Warning signs:** `"themes"` key missing from root of app.pen.

### Pitfall 4: Component ID Collisions
**What goes wrong:** Manually created IDs clash, causing ref resolution failures.
**Why it happens:** Direct JSON editing without ID generation.
**How to avoid:** Use Pencil MCP `batch_design` which auto-generates unique IDs.
**Warning signs:** `ref` nodes not rendering or showing wrong content.

### Pitfall 5: HSL-to-Hex Conversion Drift
**What goes wrong:** CSS custom properties use HSL values (`200 85% 55%`) but Pencil uses hex. Manual conversion introduces rounding errors.
**Why it happens:** Different color format standards between web CSS and design tools.
**How to avoid:** Use a consistent conversion reference. The HSL values in globals.css must be converted to precise hex for Pencil variables. Document the mapping.
**Warning signs:** Side-by-side comparison shows slightly different hues.

### Pitfall 6: Pencil MCP Server Not Running
**What goes wrong:** MCP tool calls fail silently or error out.
**Why it happens:** Pencil AppImage must be running via WSLg for MCP to connect.
**How to avoid:** Before any design work, verify: `DISPLAY=:0 ~/Applications/squashfs-root/pencil --no-sandbox &` is running, and test with `get_editor_state`.
**Warning signs:** MCP tool calls return connection errors.

## Open Questions

1. **Urination/defecation color mapping**
   - What we know: CONTEXT.md doesn't specify hues for urination and defecation explicitly. The existing code uses `violet` (urination) and `stone` (defecation) colors based on variables in app.pen.
   - What's unclear: Whether violet and stone are the intended Tailwind palettes for these domains.
   - Recommendation: Use violet for urination and stone for defecation based on existing app.pen variables. These are reasonable and distinct.

2. **Pencil MCP reliability**
   - What we know: There is a reported GitHub issue about Pencil MCP server having invalid JSON schema causing tool call failures.
   - What's unclear: Whether this affects the current version (0.6.30) and which specific tools are impacted.
   - Recommendation: If MCP tools fail, fall back to direct JSON editing of app.pen. The format is well-documented and the existing file provides a template.

3. **Component variant representation**
   - What we know: Pencil docs mention component variants but provide limited detail on how they work structurally.
   - What's unclear: Whether variants are separate reusable components or a single component with conditional properties.
   - Recommendation: Use separate reusable components per variant (e.g., "Button / Primary", "Button / Secondary", "Button / Destructive") rather than trying to use a variant system that may not be fully documented.

4. **Loading skeleton representation**
   - What we know: Skeleton states use animated gradient pulses in code (Tailwind's `animate-pulse` with `bg-muted` rectangles).
   - What's unclear: How to represent animation in a static design tool.
   - Recommendation: Use static gray rounded rectangles with `$muted` fill to represent skeleton placeholders. Add a text annotation noting "animate-pulse in code."

## CLAUDE.md Workflow Documentation

The following section should be added to CLAUDE.md under a new `### Pencil Design System` heading:

```markdown
### Pencil Design System

**Design-first workflow:** Each phase that changes UI starts with Pencil design updates, reviewed before coding.

- All designs live in `design/app.pen` (single file, version-controlled)
- Design tokens in Pencil mirror CSS custom properties in `globals.css`
- Pencil requires running AppImage: `DISPLAY=:0 ~/Applications/squashfs-root/pencil --no-sandbox &`

**Naming conventions:**
- Screens: `Screen / {PageName}` (e.g., `Screen / Dashboard`)
- Screen states: `Screen / {PageName} {State}` (e.g., `Screen / Dashboard Empty`)
- Components: `{Category} / {Name}` (e.g., `Components / Button Primary`)
- Overlays: `Overlay / {Name}` (e.g., `Overlay / Edit Intake`)
- Flows: `Flow / {Name} Step {N}` (e.g., `Flow / Add Prescription Step 1`)

**Update process:**
1. Update Pencil design for the UI change
2. Verify with `get_screenshot`
3. Implement in code to match design
4. Commit both .pen and code changes together
```
