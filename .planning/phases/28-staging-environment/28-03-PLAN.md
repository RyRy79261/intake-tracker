# Plan 28-03: CI Workflow Staging Branch Support

<objective>
Update the existing CI workflow to run on pull requests targeting the staging branch in addition to main, ensuring code quality checks apply to staging-bound changes.
</objective>

<threat_model>
## Threat Model

| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-28-03-01 | Adding staging branch to CI triggers causes unnecessary CI runs on every staging push | Low | CI trigger is `pull_request` not `push` — only PRs to staging trigger CI, direct pushes do not (per D-03, direct push is allowed) |
| T-28-03-02 | CI jobs fail on staging PRs due to missing secrets (Privy, etc.) | Low | GitHub Secrets are available to all branches in the same repo; staging PRs use the same secrets as main PRs |

No high-severity threats.
</threat_model>

## Tasks

<task id="28-03-01">
<title>Add staging branch to CI pull_request trigger</title>

<read_first>
- `.github/workflows/ci.yml` (lines 1-5 for the `on` trigger configuration)
</read_first>

<action>
Edit `.github/workflows/ci.yml` to add `staging` to the `branches` array in the `pull_request` trigger.

Current (lines 3-5):
```yaml
on:
  pull_request:
    branches: [main]
```

Change to:
```yaml
on:
  pull_request:
    branches: [main, staging]
```

This ensures that PRs targeting the `staging` branch also run the full CI suite (lint, typecheck, build, tests, etc.). Direct pushes to `staging` (allowed per D-03) do NOT trigger CI — only pull requests do.
</action>

<acceptance_criteria>
- `grep 'staging' .github/workflows/ci.yml` matches on the branches line
- `grep '\[main, staging\]' .github/workflows/ci.yml` matches exactly
- YAML remains valid: the rest of the workflow file is unchanged
- `pnpm lint` still passes (no impact on linting)
</acceptance_criteria>
</task>
