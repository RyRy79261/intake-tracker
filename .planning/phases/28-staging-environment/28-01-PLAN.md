# Plan 28-01: Disable Service Worker on Non-Production Environments

<objective>
Prevent the PWA service worker from being generated and registered on staging/preview Vercel deployments by adding a VERCEL_ENV gate to the next-pwa conditional in next.config.js and a client-side guard in use-service-worker.ts.
</objective>

<threat_model>
## Threat Model

| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-28-01-01 | Service worker installs on staging before code change is deployed, caching stale content | High | This code change MUST be merged before the staging branch is first deployed; the plan order enforces this |
| T-28-01-02 | VERCEL_ENV check breaks local development (NODE_ENV=development already guards, but future config changes could bypass) | Low | The condition is `NODE_ENV === 'production' AND VERCEL_ENV !== 'preview'` — local dev has NODE_ENV=development, so the VERCEL_ENV check never executes |
| T-28-01-03 | Client-side SW registration still attempts on staging if use-service-worker.ts is not also guarded | Medium | Belt-and-suspenders: build-time disable prevents SW file generation AND client-side guard prevents registration attempt |

Mitigation for T-28-01-01 is operational (deployment order). All other mitigations are implemented in task actions.
</threat_model>

## Tasks

<task id="28-01-01">
<title>Add VERCEL_ENV gate to next-pwa conditional in next.config.js</title>

<read_first>
- `next.config.js` (lines 1-10 for current PWA conditional, line 60 for VERCEL_ENV exposure)
</read_first>

<action>
Edit `next.config.js` line 3 to add a `VERCEL_ENV` check to the existing `NODE_ENV` conditional. The current code:

```javascript
const withPWA = process.env.NODE_ENV === 'production'
  ? require('next-pwa')({
```

Change to:

```javascript
const withPWA = process.env.NODE_ENV === 'production' && process.env.VERCEL_ENV !== 'preview'
  ? require('next-pwa')({
```

This ensures `next-pwa` is only loaded (and the service worker only generated) when:
1. `NODE_ENV` is `production` (already true — excludes local dev)
2. `VERCEL_ENV` is NOT `preview` (new — excludes staging/preview deployments on Vercel)

On Vercel, `VERCEL_ENV` is a read-only system variable: `production` for production deploys, `preview` for all other branch deploys including staging. In local development, `VERCEL_ENV` is undefined, but `NODE_ENV` is `development` so the check short-circuits.

Do NOT modify the comment on lines 1-2 — it explains the ajv concern which is still valid. Add a brief inline comment explaining the VERCEL_ENV check.
</action>

<acceptance_criteria>
- `grep "VERCEL_ENV !== 'preview'" next.config.js` returns a match on the withPWA line
- `grep "NODE_ENV === 'production'" next.config.js` still present (original check preserved)
- Both conditions joined with `&&` on the same logical expression
- `pnpm build` succeeds (no syntax errors from the change)
- `pnpm typecheck` succeeds
</acceptance_criteria>
</task>

<task id="28-01-02">
<title>Add client-side staging guard in use-service-worker.ts</title>

<read_first>
- `src/hooks/use-service-worker.ts` (full file — understand the registration flow, especially the `initSW` function starting at line 188 and the `registerServiceWorker` function starting at line 57)
- `next.config.js` (line 60 for `NEXT_PUBLIC_VERCEL_ENV` exposure confirming it is available client-side)
</read_first>

<action>
Add an early return guard at the top of the `useEffect` in `use-service-worker.ts` (inside the `initSW` function, before any registration logic) that prevents service worker registration on non-production Vercel environments.

At the very beginning of the `useEffect` callback (line 179, after the browser/SW support check on lines 181-183), add:

```typescript
// Skip SW registration on non-production Vercel environments (staging/preview)
const vercelEnv = process.env.NEXT_PUBLIC_VERCEL_ENV;
if (vercelEnv && vercelEnv !== 'production') {
  // Unregister any existing SW that may have been cached from a previous deployment
  navigator.serviceWorker.getRegistrations().then(regs => {
    regs.forEach(reg => reg.unregister());
  });
  return;
}
```

Also add the same guard at the top of `registerServiceWorker` (line 57):

```typescript
// Don't register on staging/preview environments
const vercelEnv = process.env.NEXT_PUBLIC_VERCEL_ENV;
if (vercelEnv && vercelEnv !== 'production') {
  return { success: false, error: "Service worker disabled on non-production environments" };
}
```

This is the belt-and-suspenders approach: even if the build-time disable in `next.config.js` fails for any reason (e.g., custom local production build), the client will not attempt registration. It also actively unregisters any leftover service worker from a previous deployment.
</action>

<acceptance_criteria>
- `grep "NEXT_PUBLIC_VERCEL_ENV" src/hooks/use-service-worker.ts` returns at least 2 matches (one in useEffect, one in registerServiceWorker)
- `grep "non-production" src/hooks/use-service-worker.ts` returns a match (comment present)
- `grep "unregister" src/hooks/use-service-worker.ts` returns matches (both the existing unregister function and the new cleanup in useEffect)
- `pnpm typecheck` succeeds (no type errors from the new code)
- `pnpm lint` succeeds
</acceptance_criteria>
</task>
