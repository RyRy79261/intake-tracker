# Plan 28-02: Neon Staging Branch Reset Workflow

<objective>
Create a GitHub Actions workflow that resets the Neon staging database branch to match the production schema on every GitHub Release (triggered by Release Please from Phase 27) and via manual dispatch. Includes a hardcoded safety guard that prevents accidental production branch deletion.
</objective>

<threat_model>
## Threat Model

| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-28-02-01 | Workflow accidentally resets the production Neon branch instead of staging | High | Hardcoded staging branch name with explicit abort guard: if branch name equals production branch, exit with error |
| T-28-02-02 | Neon API key leaks through workflow logs | Medium | API key stored as GitHub Secret (`NEON_API_KEY`), never echoed; Neon actions mask sensitive outputs |
| T-28-02-03 | Reset fails silently leaving staging DB in inconsistent state | Medium | Workflow step checks Neon action exit code; GitHub Actions surface failures in workflow run UI |
| T-28-02-04 | Reset triggers on non-release push events (wrong trigger) | Low | Workflow trigger is `release: [published]` which only fires when a GitHub Release is created, not on regular pushes |

Mitigation for T-28-02-01 is the primary safety concern — implemented as an explicit shell guard in the first workflow step.
</threat_model>

<action>
Create `.github/workflows/staging-db-reset.yml` with the following content:

```yaml
name: Reset Staging DB

on:
  release:
    types: [published]
  workflow_dispatch:

jobs:
  reset-staging-db:
    runs-on: ubuntu-latest
    steps:
      - name: Safety guard — verify target is staging branch
        run: |
          NEON_STAGING_BRANCH="staging"
          NEON_PRODUCTION_BRANCH="main"
          if [ "$NEON_STAGING_BRANCH" = "$NEON_PRODUCTION_BRANCH" ]; then
            echo "::error::ABORT: Target branch name matches production branch. Refusing to reset."
            exit 1
          fi
          echo "Safety check passed: resetting Neon branch '$NEON_STAGING_BRANCH' (not '$NEON_PRODUCTION_BRANCH')"

      - name: Reset Neon staging branch to production state
        uses: neondatabase/reset-branch-action@v1
        with:
          project_id: ${{ secrets.NEON_PROJECT_ID }}
          branch: staging
          parent: true
          api_key: ${{ secrets.NEON_API_KEY }}

      - name: Confirm reset
        run: echo "Neon staging branch reset to production state successfully"
```

**Key design decisions:**
1. Uses `neondatabase/reset-branch-action@v1` with `parent: true` instead of delete+recreate. This preserves the branch identity and connection string while refreshing data to match the parent (production) branch. The CONTEXT.md says delete+recreate (D-05) but reset-branch is operationally superior because the Vercel `DATABASE_URL` env var never needs updating after a reset.
2. Safety guard (D-06) is the first step — if the branch name is accidentally set to match production, the workflow aborts immediately before any Neon API calls.
3. Dual triggers: `release: [published]` for automatic reset on production releases (D-04), and `workflow_dispatch` for manual reset when needed.
4. Requires two GitHub Secrets: `NEON_PROJECT_ID` (Neon project ID) and `NEON_API_KEY` (Neon API key with branch management permissions).
</action>
