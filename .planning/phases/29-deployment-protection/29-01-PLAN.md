# Plan 29-01: Create Promotion Workflow with Neon Snapshot and Environment Gate

<objective>
Create a separate GitHub Actions workflow file that gates staging-to-main promotions with a GitHub Environment approval step and creates a Neon production database snapshot before merge. This workflow is separate from ci.yml per DEP-03 and uses the existing Production environment for reviewer approval per DEP-02.
</objective>

<threat_model>
## Threat Model

| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-29-01-01 | Promotion workflow runs for non-staging PRs to main, creating unnecessary snapshots | Medium | `if: github.head_ref == 'staging'` condition on the job ensures snapshot only runs for staging-to-main PRs |
| T-29-01-02 | Neon API key leaked in workflow logs | High | Secrets are referenced via `${{ secrets.NEON_API_KEY }}` — GitHub Actions masks secret values in logs automatically. The curl command uses `-s` (silent) to suppress progress output. |
| T-29-01-03 | Snapshot API failure blocks all production promotions | Medium | `continue-on-error: true` on the snapshot step ensures promotion is not permanently blocked by a Beta API failure. A warning is emitted instead. |
| T-29-01-04 | Wrong Neon branch gets snapshot (e.g., staging branch instead of production) | High | `NEON_PROD_BRANCH_ID` secret explicitly targets the production branch. Variable name clearly indicates "prod" to prevent misconfiguration. |
| T-29-01-05 | Workflow file has syntax error preventing it from loading | Low | Acceptance criteria includes YAML syntax validation via `python3 -c "import yaml; yaml.safe_load(open(...))"` or equivalent |

All mitigations are implemented in the workflow file itself.
</threat_model>

<action>
Create `.github/workflows/promote-to-production.yml` with the following content:

```yaml
name: Promote to Production

on:
  pull_request:
    branches: [main]
    types: [opened, synchronize, reopened]

jobs:
  pre-promote:
    # Only run for staging-to-main promotions
    if: github.head_ref == 'staging'
    runs-on: ubuntu-latest
    environment: Production
    steps:
      - name: Verify promotion source
        run: |
          echo "Promotion PR from staging to main"
          echo "Source: ${{ github.head_ref }}"
          echo "Target: ${{ github.base_ref }}"
          echo "SHA: ${{ github.sha }}"

      - name: Create Neon production snapshot
        id: snapshot
        continue-on-error: true
        run: |
          SNAPSHOT_NAME="pre-promote-$(echo ${{ github.sha }} | cut -c1-7)-$(date +%Y%m%d)"
          echo "Creating snapshot: $SNAPSHOT_NAME"
          
          RESPONSE=$(curl -s -w "\n%{http_code}" -X POST \
            "https://console.neon.tech/api/v2/projects/${{ secrets.NEON_PROJECT_ID }}/branches/${{ secrets.NEON_PROD_BRANCH_ID }}/snapshots" \
            -H "Authorization: Bearer ${{ secrets.NEON_API_KEY }}" \
            -H "Content-Type: application/json" \
            -d "{\"name\": \"$SNAPSHOT_NAME\"}")
          
          HTTP_CODE=$(echo "$RESPONSE" | tail -1)
          BODY=$(echo "$RESPONSE" | head -n -1)
          
          if [ "$HTTP_CODE" -ge 200 ] && [ "$HTTP_CODE" -lt 300 ]; then
            echo "Snapshot created successfully"
            echo "snapshot_name=$SNAPSHOT_NAME" >> "$GITHUB_OUTPUT"
          else
            echo "::warning::Snapshot creation failed (HTTP $HTTP_CODE): $BODY"
            echo "::warning::Promotion will continue without snapshot. Check Neon API status."
            exit 1
          fi

      - name: Report snapshot status
        if: always()
        run: |
          if [ "${{ steps.snapshot.outcome }}" == "success" ]; then
            echo "✓ Production snapshot created: ${{ steps.snapshot.outputs.snapshot_name }}"
          else
            echo "⚠ Snapshot creation failed — promotion continuing without backup"
            echo "Consider manually creating a snapshot in the Neon dashboard before merging"
          fi
```

Key design decisions in this workflow:
1. **Triggers on `pull_request` to `main`** — runs automatically when a PR is created/updated
2. **`if: github.head_ref == 'staging'`** — only executes for staging-to-main PRs; non-staging PRs skip this job entirely
3. **`environment: Production`** — pauses for reviewer approval before running any steps (D-04)
4. **`continue-on-error: true`** on snapshot step — Neon API is Beta; don't block promotions on API failure
5. **Separate file from ci.yml** — satisfies DEP-03
6. **No deployment logic** — Vercel handles deployment on merge; this workflow only adds pre-merge safety gates
