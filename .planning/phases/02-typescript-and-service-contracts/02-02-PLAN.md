  <action>
    Create `.eslintrc.json` at the project root with the following structure:

    ```json
    {
      "extends": "next/core-web-vitals",
      "rules": {
        "no-restricted-imports": ["error", {
          "patterns": [
            {
              "group": ["@/lib/db"],
              "importNames": ["db"],
              "message": "Components must not import 'db' directly. Use hooks instead. Type imports are allowed."
            },
            {
              "group": ["@/lib/*-service", "@/lib/*-service.ts"],
              "message": "Components/pages must not import services directly. Use hooks in src/hooks/ instead."
            }
          ]
        }]
      },
      "overrides": [
        {
          "files": ["src/hooks/**/*", "src/lib/**/*"],
          "rules": {
            "no-restricted-imports": "off"
          }
        },
        {
          "files": ["src/components/debug-panel.tsx"],
          "rules": {
            "no-restricted-imports": "off"
          }
        }
      ]
    }
    ```

    **Per user decision:** debug-panel.tsx gets an explicit exception (it needs raw DB access for diagnostics).

    **Per RESEARCH.md discretionary recommendation:**
    - Use `importNames: ["db"]` on the db pattern so that `import type { IntakeRecord } from "@/lib/db"` is allowed (type imports don't create runtime coupling)
    - Block ALL imports from `*-service.ts` files in components/pages (both value and type)
    - Override for `src/hooks/**/*` and `src/lib/**/*` so hooks can import services and services can import db
    - Override for `src/components/debug-panel.tsx` for diagnostic access

    **IMPORTANT:** The ESLint rules will immediately cause lint errors on existing import violations. This is expected and intentional -- Plan 02-03 will fix all violations by refactoring imports to use hooks. For now, lint errors from boundary violations are acceptable. The purpose of this task is to ESTABLISH the rules, not to fix all violations.
