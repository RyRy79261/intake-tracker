  <how-to-verify>
    Set up `.env.local` with valid Neon Auth credentials before starting:
    ```
    NEON_AUTH_BASE_URL=<from Neon console>
    NEON_AUTH_COOKIE_SECRET=<openssl rand -base64 32>
    GOOGLE_CLIENT_ID=<optional — for Google OAuth test>
    GOOGLE_CLIENT_SECRET=<optional>
    ALLOWED_EMAILS=<your real email>
    DATABASE_URL=<Neon postgres URL, any dev branch>
    NEON_AUTH_TEST_EMAIL=<e2e test account>
    NEON_AUTH_TEST_PASSWORD=<e2e test password>
    ```

    Before starting: run the push table truncation against the dev branch ONCE:
    ```bash
    pnpm db:truncate-push
    ```
    (This is the Phase 41 rollout step per D-10.)

    Then `pnpm dev` and walk through each scenario:

    **Scenario 1 — Email sign-in (happy path)**
    1. Open http://localhost:3000/ in an incognito window
    2. Expect: middleware redirects to /auth
    3. On /auth, the "Sign In" tab is active by default
    4. Enter the whitelisted email + password, click Sign In
    5. Expect: redirect to /, dashboard renders with intake card

    **Scenario 2 — Google OAuth sign-in**
    1. Sign out first (see scenario 5)
    2. On /auth sign-in tab, click "Continue with Google"
    3. Complete Google OAuth flow
    4. Expect: redirect to /, dashboard renders
    5. (Skip this scenario if GOOGLE_CLIENT_ID is not configured — note in UAT output.)

    **Scenario 3 — Sign-up with non-whitelisted email**
    1. Sign out. On /auth, click "Sign Up" tab
    2. Enter an email NOT in ALLOWED_EMAILS, any password
    3. Click Create Account
    4. Expect: "Please contact the administrator to request access." (or equivalent friendly message)
    5. Expect: no session is created — attempting to navigate to / still redirects to /auth

    **Scenario 4 — Sign-up with whitelisted email**
    1. Use an email that IS in ALLOWED_EMAILS (could be the same as your sign-in email if that account doesn't exist yet on a fresh branch)
    2. Expect: account creates, signs in, redirects to /

    **Scenario 5 — Sign-out from settings**
    1. While signed in, go to /settings
    2. Expand Privacy & Security accordion
    3. Expect: Account section shows your email and "Signed in via Neon Auth"
    4. Expect: NO PIN setup card anywhere in the accordion
    5. Click Sign Out
    6. Expect: redirect to /auth, cannot access /, /history, /medications, /settings without signing in again

    **Scenario 6 — PIN gate absence verification**
    1. While signed in, open History drawer
    2. Expect: opens immediately — NO PIN dialog
    3. Close. Open Settings page
    4. Expect: renders immediately — NO PIN dialog
    5. Verify every accordion section renders without auth friction

    **Scenario 7 — Push re-subscription (optional but recommended)**
    1. In settings, enable Dose Reminders (if not already enabled)
    2. Expect: notification permission prompt, then re-subscribes under the new Neon Auth user ID
    3. (Advanced) Check the Neon push_subscriptions table and confirm the user_id column contains a Neon Auth ID format, not the old Privy DID format

    **Scenario 8 — API authentication works**
    1. Try an AI-powered action (food parsing, substance lookup, etc.)
    2. Expect: succeeds without any 401 errors in the browser console
    3. Network tab should show the request carrying no `Authorization` header — the cookie does the work

    **Scenario 9 — E2E suite passes**
    1. With dev server running and test user seeded:
       ```bash
       pnpm test:e2e
       ```
    2. Expect: all specs pass (dashboard, history, medications, settings)

    **Scenario 10 — Build + CI verification**
    1. `pnpm build` succeeds
    2. `pnpm lint` succeeds
    3. `pnpm exec vitest run` (full suite) succeeds

    If ANY scenario fails, report the exact failure (screenshot, console error, network tab) and do NOT approve. The plan owner will trace back to the responsible plan (41-01 through 41-05) and issue a revision.
