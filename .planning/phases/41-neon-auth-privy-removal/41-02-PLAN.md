<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| /auth page form → /api/auth/sign-in/email | Untrusted credentials cross to server via catch-all handler |
| Browser fetch → Neon Auth client | Client-side auth state managed by useSession, flows through signed cookie on response |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-41-07 | Tampering | Sign-up form bypassing whitelist | mitigate | Whitelist enforcement happens server-side in withAuth (plan 41-01 T-41-03); even if client-side check is bypassed, user cannot call protected APIs |
| T-41-08 | Information Disclosure | Error messages on failed login | mitigate | Neon Auth returns generic "Invalid credentials" — do not leak whether email exists |
| T-41-09 | Spoofing | Google OAuth redirect | mitigate | Neon Auth validates the OAuth state parameter internally; callbackURL is same-origin |
| T-41-10 | Elevation of Privilege | Session token in localStorage | transfer | Neon Auth uses httpOnly signed cookies — no JS access to session tokens, handled by Better Auth |
| T-41-11 | Cross-Site Request Forgery | Sign-out button | accept | Neon Auth includes CSRF protection on mutation endpoints; single-user app low-value target |
