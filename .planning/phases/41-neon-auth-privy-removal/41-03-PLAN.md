<objective>
Atomically truncate the three push tables (per D-10, D-11) and strip Bearer-token plumbing out of the client push code paths (per D-06). After this plan, the first successful Neon Auth sign-in re-registers the device's push subscription under the new user ID via the existing `useDoseReminderToggle` / `subscribeToPush` flow.

Purpose: The push tables currently store Privy user IDs as the primary key. Once auth swaps, those IDs become orphaned — no code path will ever match them again. Rather than attempt a mapping migration (there is no way to map Privy DIDs to Neon Auth UUIDs), we wipe and let the existing re-subscription flow rebuild under the new identity. Single-user, single-device means one notification gap during the migration window — acceptable trade per D-10.

Output: Truncation script committed; push client helpers updated; `pnpm test` for the push client signature test passes.
</objective>

  <action>

    1. Create `scripts/truncate-push-tables.ts`:
       ```typescript
       /**
        * Phase 41 one-shot: wipe push tables before switching auth identity.
        *
        * Why: push_subscriptions.user_id was populated with Privy DIDs. After Neon Auth
        * swap, those DIDs become orphaned — no new Neon Auth user will ever match them.
        * Rather than attempt a mapping migration, we truncate and let the existing
        * subscribeToPush() flow re-register the device under the new identity on first
        * login (see src/hooks/use-push-schedule-sync.ts useDoseReminderToggle).
        *
        * Single-user, single-device: one notification gap during the migration is acceptable.
        *
        * Run with: pnpm tsx scripts/truncate-push-tables.ts
        * Requires: DATABASE_URL in env (same Neon connection string used by push-db.ts)
        */
       import { neon } from "@neondatabase/serverless";

       async function main() {
         const url = process.env.DATABASE_URL;
         if (!url) {
           console.error("DATABASE_URL is required");
           process.exit(1);
         }

         const sql = neon(url);

         console.log("[truncate-push] Starting atomic truncate of 3 push tables...");

         // Neon HTTP driver supports multi-statement transactions via sql.transaction([...])
         // If the driver version does not support transactions, fall back to sequential
         // TRUNCATE ... CASCADE calls — NOT atomic, but acceptable for a one-shot migration
         // on a single-user system with no concurrent writers.
         try {
           await sql.transaction([
             sql`TRUNCATE TABLE push_subscriptions RESTART IDENTITY CASCADE`,
             sql`TRUNCATE TABLE push_schedules RESTART IDENTITY CASCADE`,
             sql`TRUNCATE TABLE push_sent_log RESTART IDENTITY CASCADE`,
           ]);
         } catch (err) {
           // Fallback for drivers without sql.transaction support
           console.warn("[truncate-push] sql.transaction unavailable, running sequentially");
           await sql`TRUNCATE TABLE push_subscriptions RESTART IDENTITY CASCADE`;
           await sql`TRUNCATE TABLE push_schedules RESTART IDENTITY CASCADE`;
           await sql`TRUNCATE TABLE push_sent_log RESTART IDENTITY CASCADE`;
         }

         console.log("[truncate-push] Done. 3 tables cleared.");
       }

       main().catch((err) => {
         console.error("[truncate-push] Failed:", err);
         process.exit(1);
       });
       ```
