### 4. Seed-e2e-user (Task 4) — ✅ SKIPPED, bug discovered and fixed in commit `1f22b7d`

Account already existed in Neon Auth from Scenario 4 UI signup. Running the seed script anyway revealed a bug: it POSTed to `<NEON_AUTH_URL>/api/auth/sign-up/email` and got 404.

**Root cause:** `src/app/api/auth/[...path]/route.ts` is a **proxy** (`auth.handler()`). The only valid target for a raw sign-up POST is the local Next.js dev server — hitting Neon's hosted auth URL directly bypasses the proxy and 404s because Neon doesn't expose those paths.

**Worse:** the CI seed step in `ci.yml:163-170` was **structurally broken**. It ran BEFORE the E2E step, but Playwright's `webServer` config (`playwright.config.ts:63-78`) only starts the dev server during the E2E step. **Fix (commit 1f22b7d):**
- `scripts/seed-e2e-user.ts`: hardcode `LOCAL_DEV_SERVER = "http://localhost:3000"`, drop `NEON_AUTH_URL` env read, rewrite header comment explaining proxy semantics and why it's not a CI step
- `.github/workflows/ci.yml`: delete "Seed Neon Auth test user" step entirely, add explanatory comment. The seed is unnecessary in CI because Neon Auth is project-wide (see next section) — the test user is a one-time manual bootstrap that persists forever.

## Key architectural facts discovered

These are not documented anywhere else and the next agent must know them:

### Neon Auth is one-instance-per-project (beta limitation)

User tried to activate Neon Auth on the dev branch — Neon console rejected it. Neon Auth v0.1.0-beta.20 only supports one auth instance per project, attached to a single branch (typically main). **Implications:**
- The test user lives in the **shared project-wide Neon Auth user store**
- It persists across all Neon branches and all CI runs
- After Phase 41 merges to prod, the test user still exists there — it's real, whitelisted via `ALLOWED_EMAILS`, but harmless (no IndexedDB data on your phone).
- `NEON_AUTH_URL` in `.env.local` is **not branch-scoped** — same value works regardless of which `DATABASE_URL` branch you point at.

### `DATABASE_URL` and `NEON_AUTH_URL` are decoupled

- `DATABASE_URL` → your app's own schema (push tables, future Drizzle tables)
- `NEON_AUTH_URL` → Neon's hosted auth service (users/sessions live there, NOT in your `DATABASE_URL` schema)

Sign-in, sign-out, and session management do NOT touch `DATABASE_URL`. They go through the proxy to `NEON_AUTH_URL`. This is why a fresh dev branch with no schema can still sign users in fine.

### Vercel-Neon integration provisions three environments

| Vercel env | Neon target | When |
|---|---|---|
| Production | main Neon branch | prod domain |
| Preview | ephemeral branch per PR (auto-created, auto-deleted) | PR previews |
| Development | **not auto-provisioned** — must be created manually | local dev |

The user just created the Development branch manually. It's not tied to any PR.

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Starting dev server automatically | User feedback: always let user run `pnpm dev` themselves | advisory | Never invoke `pnpm dev` in tool calls |
| Using Haiku for user-facing AI routes | World-knowledge gaps caused Mio Mio Mate 1.9 mg/100ml vs 20 actual | blocking | All user-facing AI routes must use `CLAUDE_MODELS.quality` and include `WEB_SEARCH_TOOL` |
| `window.location.href = ...` for in-app nav | Breaks Next.js routing, full reload, state loss | blocking | Use `router.replace/refresh` from `next/navigation`; `window.location.reload()` still OK for SW updates |
| Conflating sodium (Na) and salt (NaCl) in AI responses | Salt ≈ 2.5× sodium; silent conversion causes wrong numbers | blocking | AI schema returns `value_mg` + explicit `measurement_type`; client does the conversion |
| Using `NEON_AUTH_URL` as a fetch target | It's an upstream service URL for `createNeonAuth`, NOT an HTTP endpoint the client can hit directly. `/api/auth/...` paths only exist on the Next.js proxy at `localhost:3000`. | blocking (new) | Never `fetch(NEON_AUTH_URL + "/api/auth/...")` — always `fetch("http://localhost:3000/api/auth/...")` for scripts, or use the `auth` client for in-app calls |
| CI steps ordered before `webServer` starts | Playwright's webServer config starts the app during the test step, not before. Any CI step that tries to hit `localhost:3000` before `pnpm test:e2e` will ECONNREFUSED. | blocking (new) | Don't add pre-test steps that depend on the app being up — either move to a before-test server, or do the work as part of globalSetup |
