## TL;DR

The test suite is **numerous, layered, and (mostly) thoughtful**: 195+
`*.test.{ts,tsx}` files, property-based tests in 10 modules, MSW
integration tests, mutation testing scoped to the highest-leverage
modules with a nightly ratchet, dual-timezone CI runs, ephemeral Neon
branches per E2E and per schema-migration run, and meta-tests that
lock the CI workflow shape against silent drift. The strategy in
`TESTING_STRATEGY.md` already considered most of the obvious gaps and
shipped Phase-A/B work.

The **remaining gaps cluster in three places**:

1. **Data-loss-adjacent services with no unit tests.**
   `src/lib/backup-service.ts` (751 LOC) and
   `src/lib/health-service.ts` (244 LOC) have effectively no
   coverage. So do several >200-LOC hooks (notifications,
   permissions, medication forms).
2. **The Playwright surface is narrow.** Chromium desktop only,
   `workers: 1` on CI, no auth-lifecycle spec, no record
   edit/delete/undo flow, no mobile profile, no isolation between
   the 45 tests sharing one Neon branch + one user.
3. **Security/quality tooling outside what's already hardened.** No
   Dependabot, no CodeQL/SAST, no PR-title commitlint, ESLint
   extended only with `next/core-web-vitals` + one custom rule,
   coverage thresholds aggregate-only with no per-folder floors,
   coverage job gated by paths-filter, CI only triggers on
   `pull_request`.

The audit lists 20 prioritized proposals (§4). None are urgent
production fires — this is a "raise the floor" exercise.

### Mutation testing

`stryker.conf.json` scopes to the eight files where regressions hurt
most: the four sync modules and the four MCP OAuth modules
(`oauth.ts`, `tokens.ts`, `scopes.ts`, `whitelist.ts`).
`.github/workflows/mutation.yml` runs nightly at 03:00 UTC with a

The workflow uploads the per-mutant HTML report as an artifact and
posts the score table to the run summary.

### CI guardrails specific to this app

- **Dual-timezone runs.** `pnpm test:tz:sa` and `pnpm test:tz:de` —
  catches DST and offset bugs other suites can't see. Rare in
  practice; excellent here.
- **`schema-migration` job** applies the committed Drizzle SQL to a
  fresh ephemeral Neon branch per PR. Catches Postgres-vs-schema
  drift that static parity checks can't (bad CHECK syntax,
  cross-schema FK permissions, unreachable `neon_auth.users_sync`
  target).
- **Per-run Neon branches** for `e2e` (`ci-e2e-${{ github.run_id }}`)
  and `schema-migration` (`ci-schema-${{ github.run_id }}`), with
  always-run cleanup steps.
- **Meta-tests lock CI shape:**
  `src/__tests__/ci-workflow-structure.test.ts`,
  `supply-chain-config.test.ts`,
  `benchmark-coverage-config.test.ts`,
  `playwright-config.test.ts`,
  `bundle-security.test.ts`. Block accidental config removal.
- **Bundle-security scan** runs against the actual production
  bundle, asserting no API-key patterns or dev-only surfaces leaked.
- **Supply-chain hardening** in `pnpm-workspace.yaml`:
  `minimumReleaseAge: 1440` (24-hour quarantine on new packages),
  `trustPolicy: no-downgrade`, `blockExoticSubdeps: true`,
  `auditLevel: high`, and every `ignoreCves` entry has a rationale
  comment.
- **Chaos-as-fixtures** in `e2e/chaos.spec.ts` exercise the canonical
  PWA failure modes: network drop mid-request, AI endpoint flaky
  retry, IndexedDB quota exceeded, clock jump backward, iOS 7-day
  eviction + cold-start.

### 3.3 Mutation-score gaps

| File | Score | Survivor profile |
|---|---|---|
| `sync-engine.ts` | 52.56% | 99 of 127 surviving mutants are in error/backoff/failure paths; happy-path push/pull is well-tested |
| `sync-payload.ts` | 54.46% | 46 survivors split between `StringLiteral` (error-message mutations) and `ObjectLiteral` (assertion checks `success: false` without inspecting Zod issue codes) |

Survivor archetype breakdown across the whole mutated set:

```
StringLiteral × 49       — literal strings changed, no assertion noticed
ConditionalExpression × 46 — `if` turned to `false`, code paths skipped silently
BooleanLiteral × 33      — true/false flipped
ObjectLiteral × 29       — fields removed from returned objects
BlockStatement × 10      — function bodies emptied
```

`ConditionalExpression` survivors are the most worrying — they
indicate branches that execute but aren't asserted on. No mutation
coverage at all on `health-service`, `backup-service`, any AI route,
or any hook.

`e2e/` has 8 spec files / 45 tests. Coverage is roughly 45% of
feature surface, 60% of critical paths, **0% of mobile**, **0% of auth
error states**.

Per-spec one-liners:

| Spec | What it covers |
|---|---|
| `dashboard.spec.ts` (10 tests) | Water, salt (AI parse), coffee preset, BP, weight, urination, defecation logging |
| `medications.spec.ts` (3) | AI medicine search → 6-step wizard → dose logging with inventory decrement |
| `history.spec.ts` (5) | Page load, tab switching, empty state, BP persistence, Summary chart render |
| `settings.spec.ts` (4) | Theme + day-start-hour persistence, export download, sign-out |
| `a11y.spec.ts` (5) | Axe-core wcag2a/wcag2aa scans on `/`, `/history`, `/medications`, `/settings`; MCP consent gated on `RUN_MCP_E2E` |
| `chaos.spec.ts` (6) | Network drop mid-request, flaky API retry, IDB quota, clock jump backward, iOS eviction, malformed localStorage |
| `sync-engine.spec.ts` (6) | Push 50 records, pull server-written rows, sparse fields, SELECT chunking, offline reconnect, queue drain |
| `mcp-connector.spec.ts` (6) | OAuth metadata, consent page, DCR, full handshake → tools/list (8 tools), 401 without bearer |

**Concrete gaps:**

- **Zero auth lifecycle E2E.** No `auth.spec.ts`. Missing: sign-in
  error states (wrong password, rate limit), sign-out lifecycle,
  session expiry mid-task, whitelist denial, sign-up flow.
- **Zero mobile coverage.** `playwright.config.ts` has only
  `Desktop Chrome` (lines 47–52). No iPhone / Pixel / Mobile Safari
  project. The app is mobile-first (max-w-lg container) but the test
  matrix never sees a phone viewport.
- **No record-lifecycle E2E.** Create flows exist; edit / delete /
  undo for any record type (water, weight, BP, dose, eating,
  urination, defecation) is **never** exercised end-to-end.
- **Untested routes.** `/profile`, `/help`, `/auth/sign-up`,
  `/auth/sign-in` (errors), `/auth/reset-password`.
- **Untested features.** Push notification opt-in / permission deny /
  notification click; backup import (export is download-only); history
  chart filtering / date range; daily notes; titration plan
  interaction.
- **Service-worker behavior** is `serviceWorkers: 'block'`
  everywhere (config line 42 — intentional, makes `page.route()`
  mocks work). No opt-in spec that flips that and verifies the
  precache / offline app-shell actually works.
- **`workers: 1` on CI** (config line 25) — 45 tests run serially,
  ~5–10 min wall clock. Could be 4× faster.
- **Database not isolated per test.** One Neon branch + one auth
  user across all 45 tests. Tests can pollute each other (the
  dashboard test logs water; the history test reads it back). Blocks
  reliable test reordering.
- **HTML report not uploaded**, only raw `playwright/` trace. Harder
  to debug without the trace-viewer chrome.

### 3.5 Coverage policy gaps

- Thresholds are **aggregate only**
  (`vitest.config.ts:34–39`): lines 54, statements 53, functions 45,
  branches 44. No per-folder floors. `src/lib/sync*` deserves >80%;
  `src/lib/mcp/` deserves >85%; `src/app/api/` deserves >75%.
- The `coverage` job is **gated by paths-filter**
  (`.github/workflows/ci.yml`, `if: needs.changes.outputs.src == 'true'`).
  Docs-only PRs skip coverage entirely — baseline drift becomes
  invisible until the next src-touching PR.
- `benchmark` job is similarly gated. Performance regressions on
  non-bench-touching PRs are invisible.

### 3.8 Husky hooks

- `.husky/pre-commit` runs `pnpm test` — the **whole suite**.
- `.husky/commit-msg` runs `pnpm dlx commitlint --edit $1`.

There is no `lint-staged`, no scope filtering. The pre-commit hook
will become onerous as the suite grows; expect contributors to
`--no-verify` past it, which silently drops commitlint validation
too.

### 3.9 Test-quality smells

- **Heavy mocking in `sync-push-route.test.ts`** — Drizzle calls
  fully mocked; the test asserts route shape, not DB behavior.
  Mitigated by `sync-push-integration.test.ts` against real Postgres,
  but the unit test is misleadingly comprehensive in isolation.
- **Limited `vi.setSystemTime()` discipline.** ~13 sites across the
  whole suite. `dose-schedule-service.test.ts` pins
  `anchorTimezone: "UTC"` but doesn't mock `Date.now()` — potential
  DST-transition flake when the CI runner crosses a DST boundary at
  the wrong second.
- **Long test files** —
  `sync-engine.test.ts` (855 LOC),
  `sync-push-route.test.ts` (517 LOC),
  `inventory-service.test.ts` (500 LOC).
  Manageable but splits along behavior boundaries would improve
  readability.
- **No `.skip` / `.todo` / `.only` markers found** — hygiene is
  good.

## 4. Proposals (prioritized)

Three tiers — P0 = high leverage, low effort; P1 = structural

rationale and a rough effort estimate. Nothing here is scheduled or

   security-updates, plus the `github-actions` ecosystem. Pair with
   the existing `pnpm audit` job — Dependabot opens the PRs, audit

3. **Add a mobile Playwright project** to `playwright.config.ts` —
